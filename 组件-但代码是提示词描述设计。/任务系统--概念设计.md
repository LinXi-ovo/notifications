# 任务系统 — 概念设计

> 基于 PLAN-HUMAN.md 的需求梳理和抽象，定义核心概念、数据模型和视图结构。
> 待审阅确认后再进入实现阶段。

## 一、核心概念

### 领域对象

```
┌─────────────────────────────────────────────────┐
│                    Mission (任务/项目)              │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │ TaskNode │◄──│ TaskNode │◄──│ TaskNode │     │
│  │  (角色A)  │   │  (角色B)  │   │  (角色C)  │     │
│  └──────────┘   └──────────┘   └──────────┘     │
│       │              │              │            │
│       ▼              ▼              ▼            │
│  ┌──────────┐   ┌──────────┐                    │
│  │ TaskNode │◄──│ TaskNode │                    │
│  │  (角色A)  │   │  (角色B)  │                    │
│  └──────────┘   └──────────┘                    │
│       │                                           │
│       ▼                                           │
│  ┌──────────┐                                    │
│  │ TaskNode │                                    │
│  │  (角色C)  │                                    │
│  └──────────┘                                    │
│                                                   │
│  角色: A 👤张三, 👤李四  │  B 👤王五  │  C 👤赵六   │
└─────────────────────────────────────────────────┘
```

| 概念 | 英文标识 | 定义 |
|---|---|---|
| **任务/项目** | `Mission` | 一个完整的任务项目，包含图结构、角色定义、参与者。是顶层容器。 |
| **任务节点** | `TaskNode` | 有向无环图（DAG）中的一个节点，是具体的工作单元。包含标题、内容、负责人角色、状态、依赖信息。 |
| **依赖边** | `Edge` | 节点间的偏序关系，`source → target` 表示 target 依赖 source。所有边构成一个偏序集（哈斯图）。 |
| **角色** | `Role` | 职能角色定义（如"审批人""执行人""审核员"），可被用户认领。 |
| **角色认领** | `Assignment` | 用户 ↔ 角色 ↔ Mission 的三元组绑定。一个角色在同一个 Mission 中只能被一人认领，一人可认领多个角色。 |
| **任务状态** | `TaskStatus` | 节点的生命周期状态（见下文）。 |
| **参与者** | `Participant` | 参与该 Mission 的用户（通过认领角色加入）。 |

### 角色与视图

- **整体图**：展示 Mission 的全量 DAG，所有节点和边，按角色着色区分。
- **角色视图**：仅展示当前角色相关节点 + 它们的上下游邻居（一阶邻域），其他节点折叠/淡化。支持切换角色过滤。
- **认领机制**：用户进入 Mission → 查看可用角色列表 → 认领 → 获得该角色视图 + 操作权限。

### 哈斯图（偏序集）映射

任务依赖关系就是 **偏序关系**：
- 反自反：节点不依赖自己
- 反对称：不会出现 A→B 且 B→A（不允许循环依赖）
- 传递约简：只保留直接依赖（传递边自动推导，不存储），形成哈斯图

```
直接依赖（存储）：  A → B,   B → C,   A → D,   D → E
传递推导（按需）：  A ⟹ C  (因为 A→B→C)
哈斯图渲染：        A ──→ B ──→ C
                    └──→ D ──→ E
```

数据只存储 **直接依赖**（`Edge` 表），渲染时按需推导传递闭包用于"阻塞检测"。

## 二、数据模型

### 2.1 Mission（任务/项目）

存储方式：JSON 文件，放在 `data/missions/` 目录下，文件名 `{mission-id}.json`。

```json
{
  "id": "mission-001",
  "title": "2026 年秋季评奖评优",
  "description": "综测计算 → 材料审核 → 名单公示全流程",
  "createdBy": "张三",
  "createdAt": "2026-06-05T10:00:00.000Z",
  "updatedAt": "2026-06-05T10:00:00.000Z",
  "status": "active",
  "roles": [
    { "id": "role-reviewer", "name": "审核员", "description": "审核提交材料", "color": "#3B82F6" },
    { "id": "role-approver", "name": "审批人", "description": "最终审批通过", "color": "#EF4444" },
    { "id": "role-executor", "name": "执行人", "description": "录入数据、整理表格", "color": "#10B981" },
    { "id": "role-observer", "name": "观察员", "description": "仅查看，不需操作", "color": "#9CA3AF" }
  ],
  "nodes": [ /* TaskNode[] 见下 */ ],
  "edges": [  /* Edge[]    见下 */ ],
  "assignments": [
    { "roleId": "role-reviewer", "userId": "zhangsan" },
    { "roleId": "role-approver", "userId": "lisi" }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 唯一标识，`mission-{uuid短尾}` |
| `title` | string | 任务项目标题 |
| `description` | string | 概述 |
| `status` | `'active'\|'archived'\|'completed'` | 项目整体状态 |
| `roles` | Role[] | 该项目下可认领的角色定义 |
| `nodes` | TaskNode[] | 所有任务节点 |
| `edges` | Edge[] | 所有直接依赖关系 |
| `assignments` | Assignment[] | 角色认领记录 |

### 2.2 TaskNode（任务节点）

```json
{
  "id": "node-001",
  "title": "收集综测材料",
  "description": "通知各班提交综测证明材料",
  "content": "<p>请各班班长于6月15日前提交...</p>",
  "contentType": "html",
  "assignedRole": "role-executor",
  "status": "completed",
  "priority": 2,
  "deadline": "2026-06-15",
  "position": { "x": 200, "y": 100 },
  "tags": ["综测", "材料收集"],
  "attachments": [
    { "name": "综测模板.xlsx", "url": "https://cos.example.com/..." }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 节点唯一标识，`node-{uuid短尾}` |
| `title` | string | 节点标题 |
| `description` | string | 简要说明 |
| `content` | string | 富文本内容（HTML 或纯文本，复用 WgEditor） |
| `contentType` | `'html'\|'text'\|'prompt'` | 内容类型。`prompt` 表示该内容是给 AI 的提示词 |
| `assignedRole` | string | 负责该节点的角色 ID（对应 roles[].id） |
| `status` | TaskStatus | 当前状态 |
| `priority` | number | 0-3，同 Notification 优先级 |
| `deadline` | string\|null | ISO 日期 |
| `position` | `{x:number, y:number}` | 在图中的坐标位置（供渲染布局用） |
| `tags` | string[] | 标签 |
| `attachments` | Attachment[] | 附件列表 |

### 2.3 Edge（依赖边）

```json
{
  "id": "edge-001",
  "source": "node-001",
  "target": "node-002",
  "label": "需材料齐备后才能开始审核"
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 边唯一标识 |
| `source` | string | 前置节点 ID（必须先完成的） |
| `target` | string | 后置节点 ID（等待 source 完成） |
| `label` | string\|null | 依赖说明 |

**约束**：所有 Edge 构成的图必须是 **有向无环图（DAG）**，不允许循环依赖。

### 2.4 TaskStatus（状态枚举）

```
┌──────────┐    ┌────────────┐    ┌──────────┐
│   todo   │───►│ in-progress │───►│ completed │
│ (未开始)  │    │  (进行中)   │    │  (已完成)  │
└──────────┘    └────────────┘    └──────────┘
     │                │
     ▼                ▼
┌──────────┐    ┌────────────┐
│  blocked │    │ cancelled  │
│  (阻塞)   │    │  (取消)    │
└──────────┘    └────────────┘
```

| 状态 | 值 | 说明 |
|---|---|---|
| 未开始 | `'todo'` | 初始状态，前置依赖未全完成 |
| 进行中 | `'in-progress'` | 已认领并开始执行 |
| 已完成 | `'completed'` | 节点完成 |
| 阻塞 | `'blocked'` | 依赖的前置节点未完成或外部因素阻塞 |
| 取消 | `'cancelled'` | 该节点不再需要执行 |

**状态转换规则**：
- `todo` ↔ `in-progress`（双向）
- `in-progress` → `completed`（单向）
- `todo` / `in-progress` → `blocked`（单向）
- `blocked` → `todo`（恢复）
- `any` → `cancelled`（单向终结）

### 2.5 Assignment（角色认领）

```json
{
  "roleId": "role-reviewer",
  "userId": "zhangsan",
  "assignedAt": "2026-06-05T10:30:00.000Z"
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `roleId` | string | 对应 roles[].id |
| `userId` | string | 用户标识（目前用 username） |
| `assignedAt` | string | 认领时间 |

## 三、数据存储策略

采用 **纯文本 JSON + 版本控制** 的策略（对应 PLAN-HUMAN 中"三"的建议）：

```
data/missions/
├── mission-001.json       # 每个 mission 一个文件
├── mission-002.json
└── index.json             # Mission 列表索引（id → title + status 的概要）
```

- 不存 Bmob，不存关系型数据库
- JSON 文件直接用 Git 管理，diff 友好
- 运行时读取 local JSON（public/ 或通过 fetch 加载）
- 写操作通过 Pinia store → 提交到 localStorage 用于持久化，未来可接入 WebDAV/Bmob 文件存储实现多端同步
- 图的 layout position 可以使用 dagre 自动布局计算，也可以手动拖拽微调

| 路径 | 名称 | 组件 | 说明 |

### 4.3 节点交互行为

| 操作 | 行为 |
|---|---|
| **点击节点** | 右侧/底部滑出详情面板，显示节点信息 + 内容预览 |
| **双击节点** | 导航到 `/mission/:id/node/:nodeId` 详情页 |
| **拖拽节点** | 更新 position，实时保存（localStorage） |
| **缩放** | 滚轮缩放，默认适配屏幕 |
| **平移** | 拖拽空白区域平移画布 |
| **悬停边** | 显示依赖说明 label |
| **右键节点** | 快捷操作菜单（变更状态、跳转详情） |

### 4.4 节点样式规则

```
每个节点 = 卡片外观，包含：

┌──────────────────┐
│ 角色色带 (顶部 4px) │
│ 图标 + 标题        │
│ 负责人: 张三       │
│ 状态标签 + 截止日期 │
│ ───────────────── │
│ 内容预览 (2行) ... │
│ ───────────────── │
│ ⏱ 进度指示器      │
└──────────────────┘

状态颜色：
  todo      → gray/浅灰
  in-progress → blue/蓝色
  completed → green/绿色
  blocked   → red/红色 + 警告图标
  cancelled → gray/灰色 + 删除线

角色着色：
  每个角色在 roles 中定义 color
  色带 + 边框颜色 使用该色
```

## 五、核心交互流程

### 5.1 认领角色流程

```
进入 Mission 页面
    │
    ▼
查看角色列表 (roles 中未被认领的)
    │
    ▼
选择角色 → 确认认领
    │
    ▼
写入 assignments → 视图自动切换为角色视图
    │
    ▼
角色视图中高亮该角色节点 → 可操作
```

### 5.2 状态变更流程

```
用户点击节点上的"开始执行"
    │
    ├─→ 检查前置依赖是否全部 completed
    │      ├─ 是 → status → in-progress
    │      └─ 否 → 提示"等待前置任务完成" → 自动设置 blocked
    │
    ▼
用户点击"标记完成"
    ├─→ 检查 content 是否已填写
    │      ├─ 是 → status → completed
    │      └─ 否 → 提示"请先填写内容"
    │
    ▼
触发下游节点的状态重算：
  - 下游节点的所有前置依赖都 completed → 通知"可以开始"
  - 等待中 → 保持 blocked 状态
```

### 5.3 内容引用通知

TaskNode 的 content 字段可以引用现有 Notification：
- 引用方式：`[[notif:{{notificationId}}]]` 或通过选择器从通知列表中选取
- 渲染时：解析为通知标题 + 链接，点击跳转到通知详情页
- 也可以将 Notification 的内容直接复制/导入为 HTML 到 content 中

## 六、与现有系统的关系

| 维度 | 说明 |
|---|---|
| **复用 Notification 内容模型** | TaskNode.content 复用 WgEditor 编辑的 HTML 内容，可直接引用 Notification |
| **独立数据通道** | Mission 数据存 JSON 文件，不走 Bmob（区别于通知的 Bmob 存储）|
| **独立路由** | `/missions/*` 命名空间，不影响现有通知路由 |
| **用户系统复用** | 复用现有 `useUserStore` 获取用户信息和登录状态 |
| **组件复用** | WgEditor（富文本编辑）、PdfPreview、Lightbox 等可直接复用 |
| **角色与权限分离** | 任务角色 ≠ 系统管理员角色。任务角色仅在该 Mission 内有效 |

## 七、技术选型考虑

| 需求 | 候选方案 | 倾向 |
|---|---|---|
| **图渲染** | vis-network, cytoscape.js, dagre-d3, custom Canvas/SVG | **dagre** (自动布局) + **vis-network** (交互) 组合，或 **custom SVG** |
| **缩放平移** | zoom/pan 内建于库或 hammer.js | 随图库自带 |
| **自动布局** | dagre（DAG 布局算法） | dagre（与 vis-network 配合好） |
| **存储** | JSON 文件 + localStorage 缓存 | JSON 文件 |
| **编辑** | JSON 手写 或 图编辑器拖拽创建 | 阶段性：先手写 JSON 加载，后加拖拽编辑 |
