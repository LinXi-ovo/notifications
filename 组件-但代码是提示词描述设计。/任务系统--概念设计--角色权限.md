# 任务系统 — 概念设计 · 第二版

> 基于 PLAN-HUMAN.md 的需求梳理，覆盖第一版（核心哈斯图任务流）和第二版（CRUD/导入导出/自定义字段/嵌套跳转/权限状态/评论催促/角色认领审核/节点级操作权限/口令委派）全部设计。

### 设计决策（已确认）

| 问题 | 决策 |
|---|---|
| 子图嵌套深度 | **3 层** |
| 评论格式 | **Markdown**（性价比最优：比纯文本丰富，比富文本轻量，Git diff 友好） |
| 自定义字段类型 | **足够**，暂不增减 |

## 一、核心概念（第一版 + 第二版统一）

### 1.1 领域全景

```
┌────────────────────────────────────────────────────────────┐
│                       Mission (任务/项目)                     │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │ TaskNode │◄───│ TaskNode │◄───│ TaskNode │  ← DAG       │
│  │  (角色A)  │    │  (角色B)  │    │  (角色C)  │    (哈斯图)  │
│  └──────────┘    └──────────┘    └──────────┘              │
│       │                                                     │
│       ▼   ┌──────────────┐                                  │
│  ┌──────────┐ │  Sub-Mission  │  ← 嵌套                      │
│  │ TaskNode │──┤  (子 DAG)    │    (第二版)                  │
│  │  (角色A)  │ │  ┌──┐ ┌──┐   │                              │
│  └──────────┘ │  │..│ │..│   │                              │
│               │  └──┘ └──┘   │                              │
│               └──────────────┘                              │
│                                                              │
│  自定义字段: [选择框] [文本] [日期] [用户]...  (第二版)       │
│  评论串: ── 张三: 已提交 ── 李四: 收到  (第二版)            │
│  催促提醒: ⏰ 截止前 24h 自动提醒            (第二版)         │
│                                                              │
│  角色: A 👤张三  │  B 👤王五  │  C 👤赵六                   │
│  权限: 执行→编辑  │  审核→批准  │  审批→发布  (第二版)      │
└────────────────────────────────────────────────────────────┘
```

| 概念 | 版本 | 定义 |
|---|---|---|
| **Mission** | v1 | 顶层任务项目容器，包含图结构、角色定义、参与者。 |
| **TaskNode** | v1 | DAG 节点 = 具体工作单元 + 富文本内容 + 负责人角色。 |
| **Edge** | v1 | 偏序依赖边，所有边构成一个有向无环图（哈斯图）。 |
| **Role** | v1 | 职能角色定义，可被用户认领。含认领策略（审核/口令/委派）。 |
| **TaskStatus** | v1 | 节点生命周期状态。 |
| **Assignment** | v1 | 用户↔角色↔Mission 的三元组绑定，含审批状态。 |
| **CustomField** | **v2** | 自定义字段模式定义（附加在 Mission 上）。 |
| **CustomValue** | **v2** | 自定义字段的值（附加在 TaskNode 上）。 |
| **SubMission** | **v2** | 嵌套子 DAG，一个 TaskNode 内展开为下一级任务图。 |
| **JumpLink** | **v2** | 跨 Mission / 跨 Node 的跳转引用。 |
| **Comment** | **v2** | 节点级评论/备注线程（Markdown 格式）。 |
| **Reminder** | **v2** | 时间或事件触发的催促/通知规则。 |
| **Permission** | **v2** | 角色在状态转换 + 节点级操作上的权限定义。 |
| **ClaimPolicy** | **v2** | 角色认领策略（自由/审核/口令/委派），见 §2.4。 |
| **NodePermission** | **v2** | 节点级操作白名单（哪些角色/用户可以操作该节点），见 §2.12。 |

### 1.2 哈斯图映射（复习）

任务依赖关系即为 **偏序关系**，数据只存储 **直接依赖**（`Edge` 表），渲染时推导传递闭包用于阻塞检测。

```
直接依赖（存储）：  A → B,   B → C,   A → D,   D → E
传递推导（按需）：  A ⟹ C  (因为 A→B→C)
哈斯图渲染：        A ──→ B ──→ C
                    └──→ D ──→ E
```

### 2.3 Edge（依赖边）

```json
{
  "id": "edge-001",
  "source": "node-001",
  "target": "node-002",
  "label": "需材料齐备后才能开始审核"
}
```

约束：所有 Edge 构成 **有向无环图（DAG）**，不允许循环依赖。

### 2.4 Role（角色）

```json
{
  "id": "class-monitor",
  "name": "班长",
  "description": "负责收集和提交全班材料",
  "color": "#F59E0B",
  "emoji": "👨‍🎓",
  "maxAssignees": 5,
  "claimPolicy": {
    "type": "approval",
    "approverRole": "leader",
    "password": null
  }
}
```

| 字段 | 说明 |
|---|---|
| `maxAssignees` | 该角色允许多人认领（如多个审核员并行工作） |
| `claimPolicy.type` | `"free"` — 任何人自由认领 / `"approval"` — 需管理员/负责人审核 / `"password"` — 输入正确口令即可认领 / `"delegated"` — 只能由管理员/负责人委派，不能自荐 |
| `claimPolicy.approverRole` | 当 type 为 `approval` 时，指定哪个角色负责审批（如 `"leader"` 或 `"admin"`） |
| `claimPolicy.password` | 当 type 为 `password` 时，预设的口令（BCrypt 哈希存储） |

**认领策略速查**：

| 类型 | 用户操作 | 后台行为 | 适用场景 |
|---|---|---|---|
| `free` | 点击认领 → 立即生效 | 直接写入 Assignment | 普通成员、执行人 |
| `approval` | 点击认领 → 等待审核 | 生成待审批记录，通知 approverRole | 班长、审核员等敏感角色 |
| `password` | 输入口令 → 匹配即生效 | 比对 password 哈希 | 线下已约定好的角色分发 |
| `delegated` | 不可自荐，只能被委派 | 仅管理员/leader 可操作 | 特殊岗位（如监察员） |

### 2.5 Assignment（角色认领）

```json
{
  "roleId": "class-monitor",
  "userId": "zhangsan",
  "assignedAt": "2026-06-05T10:30:00.000Z",
  "status": "approved",
  "approvedBy": "admin",
  "approvedAt": "2026-06-05T11:00:00.000Z"
}
```

| 字段 | 说明 |
|---|---|
| `status` | `"pending"` — 待审核 / `"approved"` — 已通过 / `"rejected"` — 已拒绝 |
| `approvedBy` | 审批人的 userId（仅 approval 类型） |
| `approvedAt` | 审批时间 |

- `free` / `password` 类型：认领时直接设为 `approved`
- `approval` 类型：认领时设为 `pending`，审批通过才改为 `approved`
- `delegated` 类型：管理员直接创建 Assignment，状态为 `approved`

### 2.8 SubMission（嵌套 DAG — 第二版）

一个 TaskNode 可以展开为一个完整的子 Mission（即嵌套的 DAG）。

```json
{
  "id": "sub-001",
  "parentNodeId": "node-004",
  "parentMissionId": "mission-001",
  "nodes": [ /* TaskNode[] — 子 DAG 的节点 */ ],
  "edges": [ /* Edge[] — 子 DAG 的边 */ ],
  "roles": [ /* Role[] — 可继承父级或重新定义 */ ]

```

**渲染行为**：
- 父节点卡片右下角显示 🔽 嵌套标记
- 点击 🔽 展开为子图（递归展开，最多 3 层）
- 子图有独立的缩放/平移状态
- 面包屑导航：`顶层 > 综测计算 > 数据录入 > 数据校验`

### 2.12 Permission（状态转换 + 节点操作权限 — 第二版）

权限系统分为两层：**角色级默认权限** + **节点级覆盖权限**。

#### 角色级默认权限

```json
{
  "roleId": "executor",
  "transitions": [
    { "from": "todo", "to": "in-progress", "label": "开始执行" },
    { "from": "in-progress", "to": "completed", "label": "标记完成" },
    { "from": "blocked", "to": "in-progress", "label": "解除阻塞" }
  ],
  "canComment": true,
  "canEditContent": false
}
```

#### 节点级覆盖权限

当 `allowedOperators` 不为空时，只有白名单内的角色可以对该节点执行状态变更。

```json
{
  "nodeId": "node-submit-form",
  "allowedRoles": ["class-monitor"],
  "allowedUsers": [],
  "transitions": [
    { "from": "in-progress", "to": "completed", "label": "确认交表" }
  ]
}
```

**权限判定逻辑（伪代码）**：

```
function canOperateNode(user, node, mission):
  // 1. 管理员永远可以
  if user.isAdmin: return true

  // 2. 检查节点级白名单（allowedOperators / allowedUsers）
  if node.allowedOperators.length > 0:
    if user.role not in node.allowedOperators: return false
  if node.allowedUsers.length > 0:
    if user.id not in node.allowedUsers: return false

  // 3. 检查角色级状态转换权限
  perm = mission.permissions.find(p => p.roleId == user.claimedRole)
  if not perm: return false
  return perm.transitions.some(t => t.from == node.status)
```

#### 默认权限表

| 角色 | 状态转换 | 评论 | 编辑内容 |
|---|---|---|---|
| 执行人 | `todo→进行中`, `进行中→已完成`, `阻塞→进行中` | ✅ | ✅ |
| 审核员 | `已完成→已验证`, `todo→进行中` | ✅ | ❌ |
| 审批人 | `已验证→已批准`, `已批准→已发布` | ✅ | ❌ |
| 管理员 | 全部 | ✅ | ✅ |
| 观察员 | 无 | ✅ | ❌ |

#### 典型场景：交表任务只有班长能打钩

```
// Role 定义
{ id: "class-monitor", name: "班长", claimPolicy: { type: "approval", approverRole: "leader" } }
{ id: "member", name: "普通成员", claimPolicy: { type: "free" } }

// TaskNode 定义
{
  id: "node-submit",
  title: "提交班级汇总表",
  assignedRole: "class-monitor",
  allowedOperators: ["class-monitor"],  // ← 只有班长能操作
  status: "in-progress"
}

// 效果：
// 普通成员可以看到这个节点，但"变更状态"按钮被禁用
// 班长可以看到且可以点击"确认交表"完成该节点
```

```
方式 A: 自动规则                方式 B: 手动催促

### 4.6 角色认领流程

```
方式 A: 自由认领 (free)           方式 B: 审核认领 (approval)
─────────────────────            ─────────────────────
进入 Mission → 角色列表           进入 Mission → 角色列表
    │                                 │
    ▼                                 ▼
点击"认领"                        点击"认领"
    │                                 │
    ▼                                 ▼
立即生效 → Assignment(status=approved)  创建待审批记录 (status=pending)
    │                                 │
    ▼                                 ▼

                                         │
                                         ▼
                                  approver 打开审批列表
                                         │

                                    │        通知认领人
                                    │
                                    └→ 拒绝 → status=rejected
                                             可填写拒绝理由

## 五、状态管理（Pinia Store）

```
missions/
├── mission.js            # 核心 store (加载/切换/CRUD)
├── mission-graph.js      # 图状态 (当前选中节点/边/缩放)
├── mission-comment.js    # 评论管理 (第二版)
└── mission-reminder.js   # 提醒管理 (第二版)
```

### useMissionStore
- `missions` — Mission 列表
- `currentMission` — 当前加载的 Mission
- `loadMission(id)` — 加载单个 Mission JSON
- `createMission(data)` — 新建
- `updateMission(patches)` — 局部更新 + 自动保存
- `deleteMission(id)` — 删除
- `addNode(data)`, `removeNode(id)`, `updateNode(id, patch)`
- `addEdge(src, tgt, label)`, `removeEdge(id)`
- `assignRole(roleId, userId)`, `unassignRole(roleId)`
- `importMission(json)`, `exportMission()` → JSON blob
- `getNodeById(id)` — 获取节点（含递归子图查找）
- `getClaimPolicy(roleId)` — 获取角色认领策略
- `claimRole(roleId)` — 认领角色（内部根据 claimPolicy 决定直接通过还是 pending）
- `claimRoleWithPassword(roleId, password)` — 口令认领
- `approveClaim(assignmentId)` — 审批通过认领
- `rejectClaim(assignmentId, reason)` — 拒绝认领
- `delegateRole(roleId, userId)` — 委派角色（仅 admin/leader）
- `getPendingClaims()` — 获取待审批的认领列表
- `checkNodeOperation(user, node)` — 判定用户是否有权操作节点

### useMissionGraphStore
- `selectedNodeId` / `selectedEdgeId`
- `activeRoleFilter`
- `zoomLevel` / `panOffset`
- `expandedSubNodes` — 当前展开的子图节点 ID 集合
- `selectNode(id)`, `selectEdge(id)`
- `setRoleFilter(roleId)`
- `toggleSubGraph(nodeId)`

### useMissionCommentStore (v2)
- `comments` — Map<nodeId, Comment[]>
- `loadComments(nodeId)`
- `addComment(nodeId, content, parentId?)`
- `updateComment(commentId, content)`
- `deleteComment(commentId)`

### useMissionReminderStore (v2)
- `reminders` — Reminder[]
- `addReminder(rule)`, `removeReminder(id)`
- `checkReminders()` — 定时检查触发条件
- `dismissReminder(id)`

## 六、与现有系统的关系

| 维度 | 说明 |
|---|---|
| **复用 WgEditor** | TaskNode.content 直接使用 WgEditor 编辑 HTML 内容 |
| **复用 Notification 引用** | content 中 `[[notif:id]]` 渲染为通知链接 |
| **独立数据通道** | Mission 数据存 JSON 文件，不走 Bmob |
| **独立路由** | `/missions/*` 命名空间，不影响现有通知路由 |
| **复用用户系统** | `useUserStore` 用于角色认领和操作者记录 |
| **复用附件模型** | Attachment 结构与通知的 File 注册表兼容 |
| **角色 ≠ 系统权限** | 任务角色仅在该 Mission 内有效，不绑定系统管理员 |

## 八、未决设计问题

1. **子图数据存储**：子图与父图同文件，还是独立文件？→ 倾向小图同文件（≤50节点），大图独立文件
2. **评论通知方式**：应用内通知栏（Pinia/event bus）vs 外部推送？→ 初期应用内通知，后续接入 Bmob 推送
3. **角色认领冲突**：多人认领同一角色时，节点操作是否需要互斥？→ 同一个节点同一时间只能一人操作（乐观锁）
4. **导入冲突策略**：ID 冲突时默认"跳过"还是"覆盖"？→ 默认跳过，用户可选覆盖
5. **提醒触发频率**：定时轮询（cron-like）还是事件驱动？→ 初期事件驱动（状态变化时检查），后续加定时扫描
